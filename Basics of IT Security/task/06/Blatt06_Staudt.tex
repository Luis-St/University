\chapter*{Aufgabe 1: Einmalpasswörter (TAN-Liste)}

\section*{a) Sicherheit bei Mithören eines genutzten Passworts}

Jedes Einmalpasswort ist nach einmaliger Nutzung entwertet.
Selbst wenn ein Angreifer ein bereits verwendetes Passwort mithört oder kopiert, kann er damit keine weitere Überweisung tätigen.
Der Server akzeptiert jedes Passwort nur ein einziges Mal.
Ein Replay-Angriff ist daher wirkungslos.

\section*{b) Sicherheit bei Abfangen von Datenpaketen}

Nein, das Verfahren wäre dann \textbf{nicht mehr sicher}.
Wenn der Angreifer Datenpakete abfangen (und nicht nur mitlesen) kann, könnte er folgendes tun:
\begin{itemize}
	\item Das Datenpaket mit dem gültigen Einmalpasswort abfangen, sodass es den Server nie erreicht.
	\item Das Einmalpasswort ist dann noch nicht entwertet, da der Server es nie erhalten hat.
	\item Der Angreifer kann das abgefangene Passwort nun selbst verwenden, um eine eigene Überweisung durchzuführen (Man-in-the-Middle-Angriff).
\end{itemize}

\section*{c) Gegenmaßnahme}

\begin{itemize}
	\item \textbf{Transaktionsbindung:} Das Einmalpasswort wird an die konkreten Überweisungsdaten gebunden (Empfänger, Betrag, IBAN).
	      Der Server prüft, ob das Passwort zu den eingereichten Transaktionsdaten passt.
	      Selbst wenn der Angreifer das Passwort abfängt, kann er es nicht für eine andere Überweisung verwenden.
	\item \textbf{Verschlüsselter Kanal (TLS):} Durch Ende-zu-Ende-Verschlüsselung wird das Abfangen und Manipulieren von Paketen verhindert.
	\item \textbf{Zeitbegrenzung:} Das Einmalpasswort ist nur für wenige Minuten gültig, was das Zeitfenster für einen Angriff stark einschränkt.
\end{itemize}

\section*{d) Verifikation über Bankkarte (chipTAN)}

\textbf{Funktionsweise:}
\begin{enumerate}
	\item Der Nutzer gibt eine Überweisung im Online-Banking ein.
	\item Die Überweisungsdaten (Empfänger-IBAN, Betrag) werden an die EC-Karte übertragen, entweder optisch über einen Flickercode am Bildschirm oder durch manuelle Eingabe am TAN-Generator.
	\item Die EC-Karte berechnet mit ihrem internen geheimen Schlüssel und den Überweisungsdaten ein Einmalpasswort (TAN).
	\item Der Nutzer gibt die angezeigte TAN im Browser ein.
	\item Der Server berechnet mit dem ihm bekannten Schlüssel dieselbe TAN und vergleicht.
\end{enumerate}

\textbf{Warum ist das sicher?}
\begin{itemize}
	\item Die TAN ist an die konkreten Transaktionsdaten gebunden.
	      Verändert ein Angreifer die Überweisung (z.\,B. anderen Empfänger), stimmt die TAN nicht mehr.
	\item Der geheime Schlüssel verlässt nie die EC-Karte.
	      Selbst bei kompromittiertem Computer kann der Angreifer den Schlüssel nicht extrahieren.
	\item Der Nutzer sieht die Überweisungsdaten auf dem TAN-Generator und kann Manipulation erkennen, bevor er die TAN eingibt.
\end{itemize}


\chapter*{Aufgabe 2: Challenge-Response-Verfahren}

\section*{a) Warum Challenge-Response?}

Das Verfahren wird genutzt, weil das Passwort \textbf{nie über das Netzwerk übertragen} wird.
Stattdessen wird nur der Hashwert aus Challenge und Passwort gesendet.
Ein Angreifer, der den Netzwerkverkehr mithört, sieht nur die Challenge und die Response, kann daraus aber nicht das Passwort rekonstruieren (Einwegeigenschaft des Hashes).
Außerdem ist jede Response einzigartig, da jede Challenge ein anderer Zufallswert ist.
Ein Replay-Angriff mit einer abgefangenen Response ist daher nutzlos.

\section*{b) Mehrfache Nutzung derselben Challenge}

Wenn dieselbe Challenge mehrfach verwendet wird, ist die Response bei gleichem Passwort jedes Mal identisch.
Das verringert die Sicherheit erheblich:
\begin{itemize}
	\item Ein Angreifer könnte eine korrekte Response aufzeichnen und bei der nächsten gleichen Challenge wiederverwenden (\textbf{Replay-Angriff}).
	\item Der Angreifer muss das Passwort nicht kennen, sondern nur die gültige Response zur bekannten Challenge.
\end{itemize}

\section*{c) Schutz vor Challenge-Wiederholung}

\begin{itemize}
	\item \textbf{Zufällige Challenges:} Jede Challenge wird kryptographisch zufällig generiert.
	      Bei ausreichender Länge (z.\,B. 128 Bit) ist eine Wiederholung extrem unwahrscheinlich.
	\item \textbf{Zeitstempel einbeziehen:} Die Challenge enthält einen Zeitstempel.
	      Der Server akzeptiert nur Responses zu aktuellen Zeitstempeln und lehnt veraltete ab.
	\item \textbf{Nonce-Speicher:} Der Server speichert bereits verwendete Challenges und lehnt jede erneut eingehende ab.
\end{itemize}

\section*{d) Beispielhafter Ablauf}

\textbf{Parameter:}
\begin{itemize}
	\item Hashfunktion: SHA-256
	\item Gemeinsames Geheimnis (Passwort): \texttt{passwort123}
\end{itemize}

\textbf{Ablauf:}

\begin{enumerate}
	\item \textbf{Client} sendet Login-Anfrage an den Server.
	\item \textbf{Server} generiert eine zufällige Challenge und sendet sie an den Client:
	      \begin{lstlisting}[language=bash]
Challenge = X7kP9mQ2
	      \end{lstlisting}
	\item \textbf{Client} berechnet die Response:
	      \begin{lstlisting}[language=bash]
Response = SHA-256("X7kP9mQ2" + "passwort123")
         = SHA-256("X7kP9mQ2passwort123")
         = D1A9107BFFC537598B50769DC0883506
           D2F79F0E23BA4D3F22AF005C878DD1C4
	      \end{lstlisting}
	\item \textbf{Client} sendet die Response an den Server.
	\item \textbf{Server} berechnet mit dem ihm bekannten Passwort denselben Hash und vergleicht.
	      Da beide übereinstimmen, ist die Authentifizierung erfolgreich.
\end{enumerate}

Das Passwort wurde dabei zu keinem Zeitpunkt über das Netzwerk übertragen.


\chapter*{Aufgabe 3: Challenge-Response-Authentifizierungssystem}

\section*{Beschreibung}

Das Programm \texttt{ChallengeResponse.java} implementiert ein Challenge-Response-Authentifizierungssystem mit zwei Benutzern (\texttt{alice} und \texttt{bob}).
Es verwendet SHA-256 als Hashfunktion und 128-Bit-Zufallswerte als Challenges.

\textbf{Ablauf:}
\begin{enumerate}
	\item Der Client sendet seinen Benutzernamen an den Server.
	\item Der Server prüft, ob der Benutzer existiert, und generiert eine zufällige Challenge.
	\item Der Client berechnet die Response als \texttt{SHA-256(Challenge + Passwort)} und sendet sie zurück.
	\item Der Server berechnet die erwartete Response mit dem gespeicherten Passwort und vergleicht.
\end{enumerate}

Das Programm demonstriert vier Szenarien:
\begin{itemize}
	\item Erfolgreicher Login von Alice
	\item Erfolgreicher Login von Bob
	\item Fehlgeschlagener Login (falsches Passwort)
	\item Fehlgeschlagener Login (unbekannter Benutzer)
\end{itemize}
